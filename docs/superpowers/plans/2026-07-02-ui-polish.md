# UI Polish Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Polish the wedding app's general UI — hero photo/carousel, a consistent 3-tier typography scale, a floating-action-button style "Přidat aktivitu" control, and a friendlier Lavička nápadů (idea bench) empty/subtitle state — without touching the blocks/timeline data model (see the companion plan `2026-07-02-blocks-feature.md` for that).

**Architecture:** Plain React 19 + Vite client, no CSS framework/component library, no icon or animation library, no test framework (verification is manual via the dev server, per project convention — the only "logic" here is CSS/markup, nothing warrants `node:test`). All changes are additive CSS + small component edits; no new dependencies. Typography becomes token-driven via three new CSS custom properties added to `client/src/styles/variables.css`.

**Tech Stack:** React 19 (JSX, no TypeScript), Vite, plain CSS with custom properties, native HTML5 drag-and-drop (already in place, not modified here).

**Design decisions locked in before coding (do not re-litigate mid-task):**
1. **Typography scale scope.** The spec's "Heading / Body / Caption, max 3 sizes" rule governs *app UI* text — tab/section headings, form/body copy, buttons, metadata. It does **not** apply to `Hero.jsx`'s marketing display text (`.hero-title`, `.hero-countdown-number`, `.hero-date`) — those are an intentionally oversized one-off "display" moment, exempted the same way a hero headline is exempted in any type-scale system. `GuestList.css`, `AccommodationList.css`, `GuestSummaryCard.css` belong to components that `App.jsx` never mounts (dead/unused code per the codebase survey) — they are **out of scope**; fixing invisible CSS is not testable and not requested implicitly. In-scope files: `variables.css`, `global.css`, `Hero.css` (only the non-display bits), `AgendaTimeline.css`, `TaskList.css`.
2. **Category assignment rule** (documented once here, applied mechanically in Task 2):
   - **HEADING** (20px / weight 700 / line-height 1.2) — page and section titles only: `h2`/`h3` used as a section heading (`AgendaPage`'s "Program svatebního dne", `IdeaBench`'s "🗂️ Lavička nápadů", and the `h3` titles inside `AgendaForm`/`BlockForm`/`CategoryForm` modals).
   - **BODY** (15px / weight 400 regular, 700 semibold for labels & buttons / line-height 1.5) — primary readable content and interactive control labels: form labels, `input`/`select`/`textarea` text, all `.btn`/`.seg-btn`/`.pill`/`.filter-btn`/`.bench-add-btn` button text, activity name (`.ar-name`), bench card name (`.bc-name`), block name label (`.bh-name` — per spec task 7's own words, "one degree bigger than caption" lands exactly on Body), descriptions (`.ar-desc`, `.bench-hint`, new `.bench-subtitle`), `.task-name`.
   - **CAPTION** (12px / weight 400 regular, 700 semibold for uppercase micro-labels / line-height 1.5) — times, metadata, auxiliary tags: `.ar-time`, `.as-time-start`, `.as-time-end`, `.bh-time`, `.gap-label`, `.ar-dur`, `.badge`, `.ar-cat`, `.ar-loc`, `.pg-lane-head`, `.block-chip`, `.block-preview-chip`, `.mgmt-label`, `.filter-label`, `.bc-priority`, `.bc-loc`, `.bench-group-title`, `.cat-preview-chip`, `.time-field-label`, `.form-computed-value`, `.as-block-badge`, `.task-meta`, `.task-note`, `.priority-badge`.
   - Emoji/icon glyphs (`✕`, `🗑`, `⠿`) are exempt — they're icons, not scaled text.
3. **FAB scope.** Only "+ Přidat aktivitu" becomes a FAB (fixed, bottom-right) on mobile; on desktop it stays inline in the page header but restyled as the visually dominant action (bigger shadow, hover scale). "+ Kategorie" / "+ Blok" stay as regular `.btn-outline` secondary buttons next to it on all breakpoints — three simultaneous solid pink CTAs would compete for attention, which the spec's own emphasis on *one* prominent action argues against.
4. **Lavička empty state.** `AgendaPage.jsx` currently swaps `IdeaBench` for a separate cramped `.bench-rail-empty` rail when `ideas.length === 0`, and `IdeaBench` itself has its own (currently unreachable, dead) empty-state paragraph. This plan removes the swap entirely — `IdeaBench` always renders, at full width, with an improved internal empty state — eliminating both the duplication and the "just gray italic text" look the spec calls out.
5. **Bench→program drop-zone highlight already exists and matches spec** (`IdeaBench.jsx`'s `scheduled-drop-active` → dashed rose outline + tinted background). No change needed there; Task 4 only touches the empty state and subtitle.

import { useMemo, useState } from 'react';
import './Hero.css';

function getDaysUntilWedding() {
  const today = new Date();
  today.setHours(0, 0, 0, 0);
  const wedding = new Date(2026, 7, 22); // 22. 8. 2026 (month is 0-indexed)
  wedding.setHours(0, 0, 0, 0);
  const diffMs = wedding - today;
  return Math.max(0, Math.round(diffMs / (1000 * 60 * 60 * 24)));
}

const SLIDES = [

          />
        ))}
        <div className="hero-photo-overlay" />

          <span className="hero-countdown-number">{days}</span>
          <span className="hero-countdown-label">dní do svatby</span>
        </div>
      </div>

      {SLIDES.length > 1 && (
        <div className="hero-nav">
          <button type="button" className="hero-nav-arrow" aria-label="Předchozí fotka" onClick={() => goTo(slideIndex - 1)}>‹</button>
          <div className="hero-dots" role="tablist" aria-label="Výběr fotky">
            {SLIDES.map((slide, i) => (
              <button
                key={slide.src}
